Layouts UI — 3 Approaches

How should layouts, shortcuts, and positions be structured in the prefs window?

A

Deep Accordion (recommended)

A new "Layouts" tab with two levels of expandable rows: Layout → Shortcuts → Positions list. Consistent with the existing Bindings UI pattern.

📂 Layouts tab
  ▼ Default Layout  edge: 2%, gap: 1%
    ▼ <Alt><Super>1
      • 6x4 1:1 3:4
      • 6x4 1:1 2:4  (cycle 2)
      [+ Add position]
    ▼ <Alt><Super>2
      • 6x4 4:1 6:4
    [+ Add shortcut]
  ▶ Wide Monitor Layout
  [+ Add layout]

Pros

  • Matches existing UI patterns
  • All config visible in one page
  • No navigation required

Cons

  • Can get deep/cramped with many shortcuts
B

Navigation View (drill-down)

Layouts list → click to open layout detail page → see shortcuts. Uses Adw.NavigationView for a modern GNOME feel.

📂 Layouts tab
  ▶ Default Layout →
  ▶ Wide Monitor Layout →
  [+ Add layout]

→ Click "Default Layout":
  ← Default Layout
  Name: [Default Layout]
  Edge margin: [2%]
  Cell gap: [1%]
  ▼ <Alt><Super>1 …
  ▼ <Alt><Super>2 …

Pros

  • Clean separation of concerns
  • Modern Adw pattern
  • Scales to many layouts

Cons

  • More navigation steps
  • Adw.NavigationView is newer API (GNOME 45+)
C

Inline Text Positions

Same accordion structure as A, but positions within each shortcut are a single comma-separated text field (raw gTile format) instead of individual rows.

📂 Layouts tab
  ▼ Default Layout
    ▼ <Alt><Super>1
      Positions: [6x4 1:1 3:4, 6x4 1:1 2:4______]
    ▼ <Alt><Super>2
      Positions: [6x4 4:1 6:4__________________]
    [+ Add shortcut]
  [+ Add layout]

Pros

  • Least nesting
  • Power users can edit raw gTile strings directly
  • Simplest to implement

Cons

  • gTile format not self-explanatory
  • Error-prone text editing
